# 04_IchiPos_GUI仕様書.md

## 1. 文書の位置づけ

本書は、IchiPos v2 で追加する GUI の仕様を定義する。

本書は、以下の3文書を前提とする。

- [01_IchiPos_基本仕様書.md](./01_IchiPos_基本仕様書.md)（v1 CLI の基本仕様）
- [02_IchiPos_機能設計書.md](./02_IchiPos_機能設計書.md)（v1 CLI の機能設計）
- [03_IchiPos_構造設計書.md](./03_IchiPos_構造設計書.md)（v1 CLI の構造設計）

上記3文書は v1（CLI）の CLI 仕様を定義した文書であり、**GUI追加を理由とした修正は行わない**。GUI固有の仕様は本書に記述する。

ただし、**CLI自体の仕様・挙動が変更された場合は、上記3文書へその変更を反映する**。これらの文書は「v1リリース時点の記録」ではなく、現行のCLI仕様を表す文書として維持する。実装のみを変更してドキュメントを更新しない運用は認めない（この方針は issue #36 で決定した。経緯は issue #34 を参照）。

GUI は v1 では対象外とされていた機能であり（01書 第11節、02書 第6節、03書 第21節）、本書はその v2 対応として、GUI の基本仕様・機能設計・構造設計をあわせて定義する。

---

## 2. 対象範囲

本書で扱う内容は以下とする。

- GUI の起動条件・起動モード判定
- GUI の画面構成
- GUI から見た機能仕様（画面項目・操作・処理・エラー時の扱い）
- GUI 追加に伴う内部構造（CLI とのロジック共通化方針）
- v2 GUI の対象外事項

本書では、CLI（v1）で定義済みの投稿ロジック（Misskey投稿・X投稿画面起動・画像処理等）の仕様変更は行わない。GUI は、それらのロジックを画面から利用するための入力受付／結果表示レイヤーとして追加する。

---

## 3. 概要

### 3.1 目的

GUI は、コマンドラインを使わずに IchiPos の投稿操作を行えるようにするために追加する。

### 3.2 スコープ

v2 GUI は、CLI（v1）の操作をGUI化することを目的とする。

GUI は以下を行う。

- 投稿内容の入力（直接入力／テキストファイル読み込み）
- 画像フォルダの選択・クリップボードからの画像貼り付け
- 投稿の実行
- 投稿結果・エラーの表示

GUI は以下を行わない（詳細は第10節）。

- 設定ファイル（`config.yaml`）の編集
- 投稿履歴の保存・表示
- 投稿後の添付画像の自動削除（G-004廃止。添付画像は投稿後もそのまま残る）

### 3.3 動作環境

GUI の動作環境は、01書 第2節の CLI 動作環境に準ずる。

| 項目 | 内容 |
|---|---|
| OS | Windows 11 |
| 言語 | C# |
| フレームワーク | .NET 10 |
| GUI技術 | WPF |
| アプリ形式 | 実行ファイル1つ（CLI/GUI 共通、詳細は第9節） |

## 4. 起動モード

### 4.1 モード判定

IchiPos は、起動時のコマンドライン引数の有無によって、CLIモードとGUIモードを自動的に切り替える。

| 条件 | モード |
|---|---|
| コマンドライン引数が1つ以上ある | CLIモード（01〜03書の仕様通り） |
| コマンドライン引数がない | GUIモード（本書の仕様） |

`--version` のみを指定した場合も「引数が1つ以上ある」ため、CLIモードとして扱う（動作は01書 第3節・02書 F-012 の通り）。

### 4.2 モードごとの独立性

CLIモードの動作は、GUI追加によって変更しない。

GUIモードは、CLIモードと同一の実行ファイルで提供する（第9章参照）。

## G-001 起動モード判定機能

### 1. 目的

起動時のコマンドライン引数の有無から、CLIモードとGUIモードを判定する。

### 2. 処理

1. コマンドライン引数を取得する。
2. 引数が1つ以上ある場合、CLIモードとして01〜03書の処理へ進む。
3. 引数がない場合、GUIモードとして起動する。

### 3. モードごとの追加処理

実行ファイルはGUIサブシステムの実行ファイルであるため（第9.2節）、GUIモードでは追加処理を行わない（コンソールはそもそも割り当てられない）。

CLIモードでは、起動直後に呼び出し元のコンソールへアタッチする（第9.2節）。

### 4. エラー条件

本機能自体はエラーを発生させない。GUIモード起動後の設定読み込みエラーは G-009 で扱う。

### 5. 出力

判定結果に応じて、CLIモードまたはGUIモードの処理へ進む。

## G-002 GUI投稿内容入力機能

### 1. 目的

投稿内容（テキスト）を画面から入力できるようにする。

### 2. 入力方法

投稿内容は、以下いずれかの方法で入力する。

- P-01 テキストボックスへの直接入力
- P-03「ファイルから読み込む...」ボタンによる `.txt` ファイルの読み込み

### 3. 直接入力

ユーザーは P-01 に投稿内容を自由に入力・編集できる。

### 4. ファイル読み込み

1. 「ファイルから読み込む...」ボタン押下時、ファイル選択ダイアログ（拡張子フィルタ `.txt`）を表示する。
2. ユーザーがファイルを選択した場合、テキストファイル読み込み機能（02書 F-003）と同じ規則でファイルを読み込む。
3. 読み込みに成功した場合、読み込んだ内容で P-01 の内容を置き換える。置き換え後、ユーザーはその内容をさらに編集できる。
4. 読み込みに失敗した場合、P-01 の内容は変更せず、エラー内容を結果ログ（G-006）に表示する。

### 5. 文字数表示

P-02 には、P-01 の現在の文字数と上限文字数（設定ファイルから取得した Misskey/X 双方の上限のうち短い方、02書 F-006 第4節と同じ算出方法）を表示する。

文字数表示は、P-01 の入力中の文字数（`{date}` 未置換の文字数）をそのまま表示する目安表示である。`{date}` を含む場合、投稿実行時に置換された後の文字数とは異なる場合がある。実際の上限チェックは、投稿実行時に置換後のテキストに対して行う（F-006・F-013 と同じ）。

### 6. 日付プレースホルダの扱い

P-01 の内容に対する `{date}` プレースホルダの置換（01書 第4.4節・02書 F-013）は、入力中には行わず、投稿実行（G-005）時に行う。

### 7. エラー条件

以下の場合はエラーとする。

- 選択した `.txt` ファイルが読み込めない。
- 選択した `.txt` ファイルのエンコードが対応外である。

（対応エンコードは01書 第4.2節と同一）

### 8. 出力

P-01 の内容を、投稿実行機能（G-005）への入力として使用する。

## G-003 GUI画像フォルダ選択機能

### 1. 目的

添付画像フォルダを画面から選択できるようにする。

### 2. 処理

1. 「参照...」ボタン押下時、フォルダ選択ダイアログを表示する。
2. ユーザーがフォルダを選択した場合、フォルダ内の対応画像形式（01書と同じ拡張子: `.png` `.jpg` `.jpeg` `.gif`）のファイルをファイル名昇順に列挙し、そのフルパスで添付画像一覧（G-011のAttachedImages、P-13）を**全て置き換える**。
3. ダイアログをキャンセルした場合、添付画像一覧は変更しない。
4. 「クリア」ボタン押下時、添付画像一覧を空にする（G-011第3節を参照）。

### 3. 画像フォルダ未選択時の扱い

添付画像一覧が空の場合、画像添付なしとして扱う（01書 第5.2節と同じ）。

### 3.1 クリップボード画像貼り付けとの関係

クリップボード画像貼り付け機能（G-010）は添付画像一覧へ**追加**するのに対し、本機能（フォルダ選択）は一覧を**全て置き換える**。フォルダ選択を行うと、それまでに貼り付けにより追加されていた一時ファイルは全て削除される（G-011第4節）。逆に、フォルダ選択後にクリップボード貼り付けを行った場合は、フォルダ選択で追加された項目はそのまま残り、貼り付け内容が追加される。

### 4. フォルダ内容の事前検証

フォルダ選択時点で存在確認・列挙は行うが、各ファイルが実際に画像として読み込み可能かどうかは事前検証しない。読み込み可否は投稿実行機能（G-005）の中で、画像添付対象判定機能（F-005 のGUI向け実装）により検証する。

### 5. エラー条件

フォルダの読み込みに失敗した場合（選択直後に外部でフォルダが削除された等）、結果ログ（G-006）にエラーを表示し、添付画像一覧は変更しない。フォルダ内の画像読み込み不可は投稿実行時（G-005）にエラーとする。

### 6. 出力

添付画像一覧（P-13）の内容を、投稿実行機能（G-005）への入力として使用する。

## G-005 GUI投稿実行機能

### 1. 目的

「投稿する」ボタン押下により、CLIと同一の投稿処理（画像一覧取得〜結果出力）を実行する。

### 2. 入力

| 項目 | 取得元 |
|---|---|
| 投稿内容 | P-01 の現在の内容 |
| 添付画像ファイルのフルパスの一覧 | 添付画像一覧（P-13）の現在の内容を追加順のまま渡す（0件の場合は画像添付なしとして扱う） |

### 3. CLIとの処理差異

GUIは、コマンドライン引数の組み立て・再解析を行わない。P-01 の値と、添付画像一覧（P-13）のファイルパスのリストを、投稿内容・添付画像として直接、投稿処理層（`IIchiPosApplication.RunAsync(content, imagePaths, config)`）に渡す。

このため、投稿内容取得機能（F-002）のうち「`<content>` が `.txt` ファイルか文字列かを判定する」規則（01書 第4.1節）はGUIには適用しない。GUIでは、P-01 の内容は常に投稿テキストの文字列そのものとして扱う。ファイルからの読み込みは、G-002 の「ファイルから読み込む」操作によって事前に行われている。

また、CLI（フォルダパス方式、F-004でフォルダ内容を読み込んでから検証）と異なり、GUIは既にフルパスが判明している画像ファイルの一覧をそのまま検証する（フォルダ結合を行わない）。これは、添付画像一覧に複数の由来（フォルダ選択由来・クリップボード貼り付け由来）が混在しうるためである（G-011参照）。

日付プレースホルダの置換（F-013）は、CLIと同じ規則で、投稿実行時に P-01 の内容に対して適用する。

### 4. 処理順序

1. 「投稿する」ボタンおよびその他の入力コントロールを無効化し、二重投稿防止機能（G-007）を作動させる。
2. 前回投稿内容と同一かどうかを判定し、同一の場合は再投稿の確認を行う（G-015）。ユーザーが再投稿しないことを選んだ場合、以降の段階へ進まず、入力コントロールを再度有効化して処理を終了する。
3. P-01 の内容に対して日付プレースホルダを置換する（F-013）。
4. 添付画像一覧（P-13）のファイルパスの一覧を対象に、画像添付対象を判定する（F-005 のGUI向け実装。フォルダ結合は行わない）。
5. 投稿前チェックを行う（F-006）。この中で、P-01への直接入力・貼り付けにより生じた末尾の空白・改行も含め、投稿テキスト末尾の空白・改行が1回だけ除去される（F-006第4.1節）。
6. Misskey に投稿する（F-007）。
7. Misskey投稿が成功した場合、X投稿画面を起動する（F-008）。
8. 添付画像が1枚以上ある場合、1枚目をクリップボードにコピーする（01書 第5.5節と同じ）。
9. 添付画像の削除は行わない（G-004廃止。CLI/GUI共有の投稿パイプライン内部では削除確認処理自体は実行されるが、GUIでは常に「削除しない」として扱うため、添付画像はそのまま残る）。
10. 投稿処理が成功した場合、今回の投稿内容を「前回投稿内容」として記録する（G-015）。
11. 各段階の結果・エラーを結果ログ（G-006）に出力する。
12. 入力コントロールを再度有効化する（成功・失敗を問わず）。

### 5. エラー時の扱い

いずれかの段階でエラーが発生した場合、CLIの終了コード相当の扱い（処理中断）を行うが、GUIではプロセスを終了しない。該当エラーをG-006で表示し、以降の段階へ進まない。ウィンドウは開いたままとする。

Misskey投稿が失敗した場合、X投稿画面は開かない（01書 第10.3節と同じ）。

X投稿画面起動が失敗した場合、Misskey投稿の取り消しは行わない（01書 第10.4節と同じ）。

### 6. 複数回投稿

GUIはCLIと異なり、1回の投稿処理後もウィンドウを閉じない。ユーザーはP-01の内容や添付画像一覧（P-13）を変更し、再度「投稿する」を押下できる。結果ログは投稿ごとに追記され、自動的にはクリアされない。

### 7. エラー条件

エラー条件は、01〜02書で定義された各機能（F-002・F-004〜F-008）のエラー条件と同一とする。

### 8. 出力

処理結果・エラーを結果表示機能（G-006）へ渡す。

## G-006 GUI結果表示機能

### 1. 目的

投稿処理の結果・エラーを画面上に表示する。CLIの結果出力機能（02書 F-010）のGUI版に相当する。

### 2. 表示先

P-09 のログ表示領域に、時系列で追記する。

### 3. 表示内容

CLIの結果出力機能（F-010）と同じ情報を表示する。

| 種別 | 内容 | 表示色 |
|---|---|---|
| 正常メッセージ | 投稿テキスト取得、添付画像数、Misskey投稿結果、X投稿画面起動結果、クリップボードコピー結果、画像削除結果 | 通常色／成功メッセージは緑 |
| 警告メッセージ | 複数ファイル貼り付け時の非対応拡張子ファイルの除外（G-010・G-011） | オレンジ |
| エラーメッセージ | エラー種別・エラー内容・対象ファイルまたは対象処理 | 赤 |

### 4. 出力しない情報

F-010 第5節と同じく、以下は表示しない。

- Misskey アクセストークン
- 設定ファイル全文
- APIレスポンスに含まれる機密情報

### 5. ログのクリア

「ログをクリア」ボタン押下時、P-09 の表示内容をすべて消去する。投稿処理の結果には影響しない。

### 6. エラー条件

本機能自体はエラーを発生させない。

### 7. 出力

なし（画面表示のみ）。

## G-007 GUI二重投稿防止機能

### 1. 目的

投稿処理の実行中に、同一内容または別内容の投稿が重複して実行されないようにする。

### 2. 処理

## G-010 GUIクリップボード画像貼り付け機能

### 1. 目的

クリップボードにコピーされた画像・画像ファイルを、Ctrl+Vにより投稿用画像として受け取れるようにする。1回の貼り付けにつき1枚のみ対応していたv2.1.0の制約を解消し、貼り付けるたびに添付画像一覧（P-13、G-011）へ**追加**する。また、エクスプローラーでの複数選択コピーのような、クリップボードに複数ファイルが入っている場合にも対応する。

### 2. 受け付け範囲

Ctrl+Vは、ウィンドウ内のどこにフォーカスがあっても受け付ける（P-01投稿内容テキストボックスにフォーカスがある場合を含む）。

### 3. 文字列貼り付けとの関係

Ctrl+V押下時、クリップボードの内容を確認する。

- クリップボードに画像またはファイル（複数可）が含まれる場合、本機能として処理し、他のコントロールへの既定の貼り付け動作は行わない。
- クリップボードに画像・ファイルのいずれも含まれない場合（文字列のみ等）、本機能は何も行わない。フォーカスのあるコントロールの既定の貼り付け動作（例：P-01への文字列貼り付け）を妨げない。

### 4. クリップボードの内容ごとの処理

クリップボードの内容によって、以下のいずれかとして扱う。

| クリップボードの内容 | 扱い |
|---|---|
| 単一のビットマップ画像（実体がクリップボード上にのみ存在する場合。例: スクリーンショットのコピー） | 4.1 |
| 1つ以上のファイル（ファイルドロップ形式。エクスプローラーでの単一・複数選択コピー等） | 4.2 |

ファイルドロップ形式が存在する場合はそれを優先し、ビットマップとしての取り扱いは行わない。

### 4.1 単一のビットマップ画像の貼り付け

1. 画像を一時ファイルとして保存する（`%TEMP%\IchiPos\<GUID>\pasted.png`。貼り付けのたびに新しいGUIDフォルダ・ファイルを使う）。
2. 保存した一時ファイルのパスを、添付画像一覧（P-13）の末尾に追加する。既存の一覧は変更しない。
3. この一時ファイルは「一時ファイル」として扱い、一覧から除去される際に実ファイルが削除される（G-011第4節）。

### 4.2 複数ファイル（ファイルドロップ形式）の貼り付け

1. クリップボード上のファイルパス一覧を取得する（1件のみの場合も本経路で扱う）。
2. 対応画像形式（`.png` `.jpg` `.jpeg` `.gif`）のファイルは、実パスのまま（コピーせず）添付画像一覧（P-13）の末尾に追加する。既存の一覧は変更しない。
3. 対応画像形式でないファイルが含まれていた場合、それらは追加せず、除外したファイル名を結果ログ（G-006）に警告として表示する（除外件数のみの表示は行わず、ファイル名を明記する）。
4. 本経路で追加した画像は「実ファイル（ユーザーの実ファイルへの参照）」として扱い、一覧から除去されても実ファイルは削除しない（G-011第4節）。

### 5. 画像フォルダ選択・クリア・投稿との関係

- 貼り付け（4.1・4.2）は添付画像一覧に**追加**する操作であり、フォルダ選択（G-003）のようにリスト全体を置き換えることはない。
- 貼り付け後にユーザーが「参照...」でフォルダを選択した場合、一覧は全て置き換えられ、それまでに貼り付けにより追加されていた一時ファイル（4.1由来）は削除される（G-003第3.1節）。複数ファイル貼り付け（4.2由来）の実ファイルは削除されない（参照のみのため）。
- 「クリア」（P-06）が押された場合、一覧の全項目を除去し、一時ファイル（4.1由来）のみ削除する（G-011第3節）。
- 貼り付けにより追加された画像を含む投稿実行（G-005）は、通常のフォルダ選択画像と同じ処理で行う。GUIは投稿後に画像を自動削除しない（G-004廃止）ため、由来を問わず投稿によって添付画像が削除されることはない。一時ファイル（4.1由来）は、次回のフォルダ選択・クリア・個別削除・アプリケーション終了時に削除される。

### 6. 投稿中の扱い

投稿処理実行中（G-007、IsBusy）は、Ctrl+Vによる貼り付け（4.1・4.2いずれも）を受け付けない。

### 7. エラー条件

一時ファイルへの保存に失敗した場合（4.1）、結果ログ（G-006）にエラーを表示し、添付画像一覧は変更しない。

### 8. 出力

添付画像一覧（P-13）の内容を、投稿実行機能（G-005）への入力として使用する。

## G-011 GUIサムネイル一覧・個別削除機能

### 1. 目的

添付画像一覧（フォルダ選択・クリップボード貼り付けにより追加された画像候補）を、サムネイルで一覧表示し、1件単位で除去できるようにする。

### 2. 画面項目

P-13 サムネイル一覧（横並び・横スクロール、フィルムストリップ風）。画像フォルダ欄の下に配置し、投稿内容欄・結果ログ欄など既存UIの縦方向の圧迫を避ける。各サムネイルには個別の削除ボタンを付す。

サムネイル画像は、`BitmapImage` の `DecodePixelWidth` を指定して軽量にデコードして生成する。

### 3. 個別削除・全削除

- 各サムネイルの削除ボタン押下時、その1件のみを添付画像一覧から除去する。
- 「クリア」（P-06）押下時、一覧の全件を除去する。

いずれの場合も、除去した項目が後述の「一時ファイル」に該当する場合は実ファイルを削除する。「実ファイル」に該当する場合は一覧から参照を外すのみで、実ファイルは削除しない。

### 4. 添付画像の由来と削除所有権

添付画像一覧の各項目は、追加経路によって以下のいずれかの由来を持ち、除去時の実ファイル削除要否が異なる。

| 由来 | 追加経路 | 一覧から除去された場合 |
|---|---|---|
| 一時ファイル（所有権あり） | クリップボード貼り付け（単一ビットマップ、G-010第4.1節） | 実ファイルを削除する |
| 実ファイル（参照のみ） | フォルダ選択（G-003）、複数ファイル貼り付け（G-010第4.2節） | 実ファイルは削除しない |

### 5. ドラッグ&ドロップによる並べ替え

サムネイル一覧内の項目は、マウスドラッグにより並べ替えられる。

1. サムネイルをドラッグし、別のサムネイルの上でドロップすると、ドラッグ元の項目がドロップ先の位置へ移動する（間の項目はその分だけ詰める）。実ファイルの移動・コピーは行わず、添付画像一覧内の順序のみを変更する。
2. 投稿処理実行中（G-007、IsBusy）は、ドラッグによる並べ替えを受け付けない。
3. 範囲外の位置へのドロップ等、無効な操作は無視し、一覧は変更しない。

これは、複数ファイルを同時にクリップボードへコピーした場合にOSが元の選択順を保持しない（クリップボードのファイル一覧は通常エクスプローラーの表示順になる）という制約を、ユーザーが貼り付け後に手動で補正できるようにするためのものである。

### 6. 添付順序

添付画像一覧の順序は、フォルダ選択時はファイル名昇順（G-003第2節）、それ以降の貼り付けによる追加は追加した順、さらに第5節の並べ替え操作があればその結果を反映した順とする。投稿実行（G-005）へは、この一覧の順序をそのまま渡す。

### 7. 上限

添付枚数の上限は設けない（Misskey API側の制約に委ねる）。

### 8. エラー条件

本機能自体はエラーを発生させない。

### 9. 出力

添付画像一覧（P-13）の内容を、投稿実行機能（G-005）への入力として使用する。

## G-015 GUI同一内容再投稿確認機能

### 1. 目的

前回投稿したものと同じ内容を、ユーザーが意図せず再度投稿してしまうことを防ぐ。全クリアを起点とした連続投稿（第7.8節）やCtrl+Enter投稿（G-014）では、投稿済みかどうかを見失ったまま再度「投稿する」を押下しうるため、投稿前に確認を挟む。

本機能は、投稿処理の実行中の重複実行を防ぐ二重投稿防止機能（G-007）とは目的・タイミングが異なる。G-007は1回の投稿処理中の多重実行を防ぎ、本機能は投稿処理と投稿処理の間での同一内容の重複投稿を防ぐ。

### 2. 判定対象

判定対象は投稿内容（P-01）の本文テキストのみとする。添付画像一覧（P-13）は判定対象に含めない。

比較に用いるテキストは、日付プレースホルダを置換し（F-013）、末尾の空白・改行を除去した（F-006第4.1節）ものとする。すなわち、実際にMisskeyへ投稿されるテキストと同じ形に揃えたうえで、完全一致（大文字小文字・全角半角・空白をいずれも区別する）で比較する。

`{date}` を含む投稿内容は、置換後の日付が異なれば同一とは判定しない（前日に投稿した内容を翌日に投稿する場合、確認は表示されない）。

### 3. 処理

「投稿する」（P-08）押下時またはCtrl+Enter押下時（G-014）、G-005第4節の日付プレースホルダ置換より前に、以下を行う。

1. 記録されている前回投稿内容を読み出す。記録がない場合（初回起動時・記録の読み出しに失敗した場合を含む）、確認を行わず投稿処理を続行する。
2. 第2節の比較で今回の投稿内容が前回投稿内容と一致しない場合、確認を行わず投稿処理を続行する。
3. 一致した場合、確認ダイアログを表示する。
   - メッセージ: 「前回と同じ内容です。再投稿しますか？」
   - ボタン: 「はい」「いいえ」の2択とし、既定の選択は「いいえ」とする（誤ってEnterキーを押した場合に投稿されないようにするため）。
4. 「はい」が選ばれた場合、投稿処理を続行する。
5. 「いいえ」が選ばれた場合、投稿処理を中止する。Misskey投稿・X投稿画面起動は行わず、投稿内容・添付画像一覧は変更しない。結果ログ（G-006）に「投稿を中止しました」を情報メッセージとして出力する。

### 4. 前回投稿内容の記録

1. 記録するのは、投稿処理が成功した（Misskey投稿が成功した）場合のみとする。投稿前チェックエラー・Misskey投稿エラーの場合は記録を更新しない。
2. 記録内容は、第2節の比較用テキストのSHA-256ハッシュ値（16進小文字表記）のみとし、投稿内容そのものは保存しない。本機能は一致判定しか必要としないため、投稿本文がディスク上に残ることを避ける。
3. 記録は1件のみ保持し、投稿成功のたびに上書きする（履歴は保持しない）。
4. 記録はアプリケーションの再起動をまたいで保持する。保存先は `%LOCALAPPDATA%\IchiPos\last_post_hash.txt` とする。設定ファイル（`config/config.yaml`）はユーザーが編集する設定であり、アプリケーションが書き換える状態の保存先としては用いない。
5. 記録の読み出し・書き込みに失敗した場合（フォルダを作成できない・ファイルが読めない等）、投稿処理そのものは妨げない。読み出し失敗時は「記録なし」として扱い、書き込み失敗時は無視して投稿成功として扱う。

### 5. 投稿処理実行中の扱い

投稿処理実行中（G-007、IsBusy）は「投稿する」（P-08）が無効化されるため、本機能の確認が二重に表示されることはない。

### 6. エラー条件

本機能自体はエラーを発生させない（第4節第5項のとおり、記録の入出力失敗はエラーとして扱わない）。

### 7. 出力

「いいえ」が選ばれた場合のみ、結果ログ（G-006）に情報メッセージ「投稿を中止しました」を出力する。それ以外の場合、本機能は出力を行わない。

## 7. 正常系シナリオ（GUI）

### 7.1 テキスト直接入力・画像なし投稿

1. ユーザーが P-01 に投稿内容を直接入力する。
2. 添付画像一覧（P-13）は未選択（空）のままとする。
3. 「投稿する」を押下する。
4. 画像添付なしとして投稿前チェックを行う。
5. Misskeyに投稿する。
6. X投稿画面を開く。
7. 結果ログに各段階の成功メッセージが表示される。

### 7.2 ファイル読み込み・画像付き投稿

1. 「ファイルから読み込む...」で `.txt` ファイルを選択し、P-01 に内容が読み込まれる。
2. 「参照...」で画像フォルダを選択する。
3. 「投稿する」を押下する。
4. 画像がMisskey Driveへアップロードされ、画像付きノートが作成される。
5. X投稿画面が開き、1枚目の画像がクリップボードにコピーされる。
6. 添付画像は削除されず、そのまま添付画像一覧に残る（G-004廃止）。

### 7.3 日付プレースホルダを含む投稿

1. P-01 に `今日は{date}です` と入力する。
2. 「投稿する」を押下する。
3. 投稿実行時に `{date}` が実行日（`YYYY/MM/DD`形式）に置換される。
4. 置換後のテキストで投稿前チェック・Misskey投稿・X投稿画面起動が行われる。

### 7.4 同一セッションでの複数回投稿

1. 1回目の投稿を実行し、成功する。
2. ウィンドウを閉じずに、P-01 の内容を書き換える。
3. 2回目の「投稿する」を押下し、投稿する。
4. 結果ログに1回目・2回目双方の結果が時系列で表示される。

### 7.5 クリップボード画像の複数回貼り付け

1. ユーザーが画像をクリップボードにコピーする。
2. ウィンドウ内（P-01にフォーカスがある状態を含む）でCtrl+Vを押下する。画像が一時ファイルに保存され、添付画像一覧（P-13）に追加される。
3. 別の画像をコピーし、再度Ctrl+Vを押下する。1枚目は置き換えられず、2枚目が一覧に追加される（合計2枚）。
4. 「投稿する」を押下すると、一覧の追加順で2枚ともMisskeyに画像付き投稿される。

### 7.6 貼り付け後にフォルダ選択でやり直す

1. Ctrl+Vで画像を貼り付ける（添付画像一覧に一時ファイルが追加される）。
2. 気が変わり、「参照...」で別の画像フォルダを選択する。
3. 貼り付けによる一時ファイルは削除され、添付画像一覧はフォルダ内の画像に置き換わる。

### 7.7 エクスプローラーからの複数ファイル貼り付けと個別削除

1. エクスプローラーで複数の画像ファイルを選択してコピーする。
2. ウィンドウ内でCtrl+Vを押下すると、対応画像形式のファイルが実パスのまま添付画像一覧に追加される。
3. 一覧にサムネイルが表示され、1件を削除ボタンで除去する。除去した画像はユーザーの実ファイルであるため削除されない。
4. 「投稿する」を押下すると、残りの画像が投稿される。

### 7.8 全クリアを起点とした連続投稿（メインユースケース）

1. 「全クリア」（P-15）を押下する。投稿内容・添付画像一覧・結果ログが全て消去され、投稿内容欄（P-01）にフォーカスが移る。
2. そのまま投稿内容を入力する（②）。
3. 必要に応じて「参照...」またはCtrl+Vで画像を添付する（③、添付しない場合は省略）。
4. 「投稿する」またはCtrl+Enterで投稿する（④）。
5. 次の投稿のため、再度「全クリア」を押下し、1.に戻る。

### 7.9 前回と同じ内容を投稿しようとする（G-015）

1. 投稿を1回実行し、成功する。
2. 全クリアを忘れたまま、あるいは同じ内容を再度入力した状態で、「投稿する」を押下する。
3. 「前回と同じ内容です。再投稿しますか？」の確認ダイアログが表示される（既定は「いいえ」）。
4. 「はい」を選ぶと、そのまま投稿処理が続行され、投稿される。
5. 「いいえ」を選ぶと投稿処理が中止され、結果ログに「投稿を中止しました」が表示される。投稿内容・添付画像一覧はそのまま残るため、内容を修正して投稿し直せる。

## 8. 異常系シナリオ（GUI）

### 8.1 投稿内容が空

1. P-01 が空の状態で「投稿する」を押下する。
2. 投稿前チェック（F-006）でエラーとなる。
3. Misskey投稿・X投稿画面起動を行わない。
4. 結果ログにエラー内容が表示される。
5. ウィンドウは開いたままとなり、入力コントロールは再度有効化される。

### 8.2 添付画像ファイルが読み込めない

（フォルダ選択・貼り付け後、外部でファイルが削除・破損した場合を想定）

1. 投稿実行時、添付画像一覧の画像添付対象判定（F-005 のGUI向け実装）でエラーとなる。
2. Misskey投稿・X投稿画面起動を行わない。
3. 結果ログにエラー内容が表示される。

### 8.3 Misskey投稿失敗

1. 「投稿する」押下後、Misskey投稿がエラー応答となる。
2. X投稿画面を開かない。
3. 結果ログにエラー内容が表示される。
4. 入力コントロールが再度有効化され、ユーザーは内容を修正して再投稿できる。

### 8.4 設定ファイル読み込み失敗

1. GUIモードで起動する。
2. 設定ファイル読み込みに失敗する。
3. エラーダイアログを表示する。
4. ダイアログを閉じるとアプリケーションが終了する。

### 8.5 複数ファイル貼り付けに非対応拡張子が混在

1. エクスプローラーで画像ファイルとテキストファイルを同時に選択してコピーする。
2. ウィンドウ内でCtrl+Vを押下する。
3. 画像ファイルのみ添付画像一覧に追加される。
4. 結果ログに、除外したファイル名を含む警告メッセージが表示される。

## 9. 構造・実装方針

### 9.1 既存構造との関係

GUI追加後も、03書 第4節で定義された5段階の処理構造（入力受付・入力解決・投稿前チェック・投稿実行・結果出力）を維持する。

CLIとGUIで異なるのは「入力受付」と「結果出力」の実装のみとする。

| 段階 | CLI | GUI |
|---|---|---|
| 入力受付 | コマンドライン引数解析（F-001） | 画面項目からの入力（G-002・G-003） |
| 入力解決〜投稿実行 | F-002〜F-009（共通処理） | 同左（共通処理をそのまま利用） |
| 結果出力 | 標準出力・標準エラー（F-010） | 画面ログ表示（G-006） |

投稿前チェック・投稿実行・投稿先固有処理（Misskey/X）の内部構造・責務分離方針は、03書 第7節・第8節〜第18節の方針をそのまま適用し、変更しない。

### 9.2 実行ファイル構成

GUIは、CLIと同一の実行ファイルとして提供する。実行ファイル・プロジェクトを分割しない。

実行ファイルはGUIサブシステムの実行ファイルとする。GUIサブシステムの実行ファイルは、OSによってコンソールが自動的に割り当てられないため、GUIモードでは何も行わなくてもコンソールウィンドウは表示されない。

CLIモードでは、起動直後に以下の順で標準出力の確保を試みる。

1. 呼び出し元が標準出力を既にリダイレクトしている場合（ファイル・パイプ等）は、それをそのまま使用する。
2. リダイレクトされていない場合、呼び出し元のコンソールへのアタッチを試みる。成功した場合、そのコンソールに標準出力・標準エラーが出力される（従来のCLIと同じ見え方になる）。
3. アタッチにも失敗した場合（呼び出し元にコンソールが存在しない場合）、新規コンソールを割り当てる。

### 9.3 入力受付の分離

GUIの入力受付（G-002・G-003）は、コマンドライン引数の文字列表現を経由せず、画面項目の値を直接、入力解決以降の共通処理に渡す。これにより、GUIが実在しないコマンドライン引数を組み立てる、といった不要な変換を行わない。

投稿内容取得機能（F-002）のうち、`<content>` がファイルパスか文字列かを判定する規則は、CLI固有の入力受付規則として扱い、GUIには適用しない（詳細はG-005第3節）。

### 9.4 結果出力の分離

結果出力は、CLIでは標準出力・標準エラーへの書き込み、GUIでは画面ログへの追記として実装するが、出力すべき情報の種類（正常時・エラー時、機密情報を出力しないという方針を含む）はCLI・GUIで共通とする（F-010・G-006）。

### 9.5 投稿先固有処理との関係

Misskey投稿・X投稿画面起動の投稿先固有処理（03書 第18節）は、GUI追加後も各投稿先の処理領域に閉じたままとする。GUI層・入力受付層に投稿先固有の分岐を持ち込まない。

## 10. v2 対象外

v2 GUIでは以下を対象外とする。

- 設定ファイル（`config.yaml`）のGUI編集・保存
- 投稿履歴の保存・表示
- エクスプローラー等の外部からのドラッグ&ドロップによる画像の新規追加（クリップボード画像の貼り付け（G-010）とは別の操作であり、対象外のまま。なお、既に追加済みのサムネイル同士をアプリ内でドラッグして並べ替える操作はG-011第5節で対応済み）
- `{date}` のリアルタイム置換プレビュー
- 複数投稿の同時実行・キューイング
- 複数ウィンドウ・複数タブ
- ダークモード等のテーマ切り替え
- UI文言の多言語化
- GUI起動後の設定ファイル再読み込み
- サブ投稿先の追加・複数管理・有効／無効切り替え（01書 第11節・03書 第17節の方針を継承）
