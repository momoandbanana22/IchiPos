# 04_IchiPos_GUI仕様書.md

## 1. 文書の位置づけ

本書は、IchiPos v2 で追加する GUI の仕様を定義する。

本書は、以下の3文書を前提とする。

- [01_IchiPos_基本仕様書.md](./01_IchiPos_基本仕様書.md)（v1 CLI の基本仕様）
- [02_IchiPos_機能設計書.md](./02_IchiPos_機能設計書.md)（v1 CLI の機能設計）
- [03_IchiPos_構造設計書.md](./03_IchiPos_構造設計書.md)（v1 CLI の構造設計）

上記3文書は v1（CLI）の記録として扱い、本書追加に伴う修正は行わない。

GUI は v1 では対象外とされていた機能であり（01書 第11節、02書 第6節、03書 第21節）、本書はその v2 対応として、GUI の基本仕様・機能設計・構造設計をあわせて定義する。

---

## 2. 対象範囲

本書で扱う内容は以下とする。

- GUI の起動条件・起動モード判定
- GUI の画面構成
- GUI から見た機能仕様（画面項目・操作・処理・エラー時の扱い）
- GUI 追加に伴う内部構造（CLI とのロジック共通化方針）
- v2 GUI の対象外事項

本書では、CLI（v1）で定義済みの投稿ロジック（Misskey投稿・X投稿画面起動・画像処理等）の仕様変更は行わない。GUI は、それらのロジックを画面から利用するための入力受付／結果表示レイヤーとして追加する。

---

## 3. 概要

### 3.1 目的

GUI は、コマンドラインを使わずに IchiPos の投稿操作を行えるようにするために追加する。

### 3.2 スコープ

v2 GUI は、CLI（v1）の操作をGUI化することを目的とする。

GUI は以下を行う。

- 投稿内容の入力（直接入力／テキストファイル読み込み）
- 画像フォルダの選択
- 投稿後の画像削除有無の事前設定
- 投稿の実行
- 投稿結果・エラーの表示

GUI は以下を行わない（詳細は第10節）。

- 設定ファイル（`config.yaml`）の編集
- 投稿履歴の保存・表示

### 3.3 動作環境

GUI の動作環境は、01書 第2節の CLI 動作環境に準ずる。

| 項目 | 内容 |
|---|---|
| OS | Windows 11 |
| 言語 | C# |
| フレームワーク | .NET 10 |
| GUI技術 | WPF |
| アプリ形式 | 実行ファイル1つ（CLI/GUI 共通、詳細は第9節） |

## 4. 起動モード

### 4.1 モード判定

IchiPos は、起動時のコマンドライン引数の有無によって、CLIモードとGUIモードを自動的に切り替える。

| 条件 | モード |
|---|---|
| コマンドライン引数が1つ以上ある | CLIモード（01〜03書の仕様通り） |
| コマンドライン引数がない | GUIモード（本書の仕様） |

`--version` のみを指定した場合も「引数が1つ以上ある」ため、CLIモードとして扱う（動作は01書 第3節・02書 F-012 の通り）。

### 4.2 モードごとの独立性

CLIモードの動作は、GUI追加によって変更しない。

GUIモードは、CLIモードと同一の実行ファイルで提供する（第9章参照）。

## G-001 起動モード判定機能

### 1. 目的

起動時のコマンドライン引数の有無から、CLIモードとGUIモードを判定する。

### 2. 処理

1. コマンドライン引数を取得する。
2. 引数が1つ以上ある場合、CLIモードとして01〜03書の処理へ進む。
3. 引数がない場合、GUIモードとして起動する。

### 3. GUIモード起動時の追加処理

実行ファイルはコンソールサブシステムの実行ファイルであるため（第9.2節）、GUIモードで起動した場合はコンソールウィンドウを表示しない。

### 4. エラー条件

本機能自体はエラーを発生させない。GUIモード起動後の設定読み込みエラーは G-009 で扱う。

### 5. 出力

判定結果に応じて、CLIモードまたはGUIモードの処理へ進む。

---

## G-002 GUI投稿内容入力機能

### 1. 目的

投稿内容（テキスト）を画面から入力できるようにする。

### 2. 入力方法

投稿内容は、以下いずれかの方法で入力する。

- P-01 テキストボックスへの直接入力
- P-03「ファイルから読み込む...」ボタンによる `.txt` ファイルの読み込み

### 3. 直接入力

ユーザーは P-01 に投稿内容を自由に入力・編集できる。

### 4. ファイル読み込み

1. 「ファイルから読み込む...」ボタン押下時、ファイル選択ダイアログ（拡張子フィルタ `.txt`）を表示する。
2. ユーザーがファイルを選択した場合、テキストファイル読み込み機能（02書 F-003）と同じ規則でファイルを読み込む。
3. 読み込みに成功した場合、読み込んだ内容で P-01 の内容を置き換える。置き換え後、ユーザーはその内容をさらに編集できる。
4. 読み込みに失敗した場合、P-01 の内容は変更せず、エラー内容を結果ログ（G-006）に表示する。

### 5. 文字数表示

P-02 には、P-01 の現在の文字数と上限文字数（設定ファイルから取得した Misskey/X 双方の上限のうち短い方、02書 F-006 第4節と同じ算出方法）を表示する。

文字数表示は、P-01 の入力中の文字数（`{date}` 未置換の文字数）をそのまま表示する目安表示である。`{date}` を含む場合、投稿実行時に置換された後の文字数とは異なる場合がある。実際の上限チェックは、投稿実行時に置換後のテキストに対して行う（F-006・F-013 と同じ）。

### 6. 日付プレースホルダの扱い

P-01 の内容に対する `{date}` プレースホルダの置換（01書 第4.4節・02書 F-013）は、入力中には行わず、投稿実行（G-005）時に行う。

### 7. エラー条件

以下の場合はエラーとする。

- 選択した `.txt` ファイルが読み込めない。
- 選択した `.txt` ファイルのエンコードが対応外である。

（対応エンコードは01書 第4.2節と同一）

### 8. 出力

P-01 の内容を、投稿実行機能（G-005）への入力として使用する。

## G-003 GUI画像フォルダ選択機能

### 1. 目的

添付画像フォルダを画面から選択できるようにする。

### 2. 処理

1. 「参照...」ボタン押下時、フォルダ選択ダイアログを表示する。
2. ユーザーがフォルダを選択した場合、P-04 に選択したフォルダパスを表示する。
3. 「クリア」ボタン押下時、P-04 の内容を空にする（画像フォルダ未選択の状態に戻す）。

### 3. 画像フォルダ未選択時の扱い

P-04 が空の場合、画像添付なしとして扱う（01書 第5.2節と同じ）。

### 4. フォルダ内容の事前検証

本機能では、選択したフォルダ内の画像有無・読み込み可否を事前検証しない。これらは投稿実行機能（G-005）の中で、画像フォルダ読み込み機能（F-004）・画像添付対象判定機能（F-005）により検証する。

### 5. エラー条件

本機能自体はエラーを発生させない。フォルダ不正・画像読み込み不可は投稿実行時（G-005）にエラーとする。

### 6. 出力

P-04 の内容を、投稿実行機能（G-005）への入力として使用する。

---

## G-004 GUI画像削除事前設定機能

### 1. 目的

投稿完了後に添付画像を削除するかどうかを、投稿前に画面上で設定できるようにする。

CLI（02書 F-011）では投稿完了後に y/n の確認を行うが、GUIでは投稿前のチェックボックスにより事前に意思決定する。

### 2. 画面項目

P-07「投稿後に画像を削除する」チェックボックス。

### 3. 初期状態

P-07 の初期状態はオフ（削除しない）とする。

### 4. 有効／無効の制御

P-04（画像フォルダパス）が空の場合、P-07 は無効化（操作不可）とし、チェック状態にかかわらず削除処理は行わない。

P-04 にフォルダパスが設定されている場合、P-07 は操作可能とする。

### 5. 処理

投稿実行機能（G-005）の中で、Misskey投稿とX投稿画面起動の両方が成功し、かつ添付画像が1枚以上存在した場合に、P-07 の状態を参照する。

- P-07 がオンの場合、添付した画像ファイルをすべて削除する。
- P-07 がオフの場合、削除を行わない。

いずれの場合も、結果を結果ログ（G-006）に出力する。

### 6. エラー条件

削除処理はCLI（F-011）と同様に投稿完了後に実行するため、削除に失敗してもメイン処理の結果には影響しない。削除失敗時はその旨を結果ログに出力する。

### 7. 出力

| 条件 | 出力内容 |
|---|---|
| 削除した場合 | 削除完了メッセージ（削除枚数を含む） |
| 削除しなかった場合（P-07オフ） | スキップメッセージ |
| 削除に失敗した場合 | 削除失敗メッセージ（メイン処理の結果には影響しない） |

## G-005 GUI投稿実行機能

### 1. 目的

「投稿する」ボタン押下により、CLIと同一の投稿処理（画像一覧取得〜結果出力）を実行する。

### 2. 入力

| 項目 | 取得元 |
|---|---|
| 投稿内容 | P-01 の現在の内容 |
| 画像フォルダパス | P-04 の現在の内容（空の場合は未指定として扱う） |
| 画像削除事前設定 | P-07 の現在の状態 |

### 3. CLIとの処理差異

GUIは、コマンドライン引数の組み立て・再解析を行わない。P-01・P-04 の値を、投稿内容・画像フォルダパスとして直接、画像一覧取得以降の処理（F-004〜F-010）に渡す。

このため、投稿内容取得機能（F-002）のうち「`<content>` が `.txt` ファイルか文字列かを判定する」規則（01書 第4.1節）はGUIには適用しない。GUIでは、P-01 の内容は常に投稿テキストの文字列そのものとして扱う。ファイルからの読み込みは、G-002 の「ファイルから読み込む」操作によって事前に行われている。

日付プレースホルダの置換（F-013）は、CLIと同じ規則で、投稿実行時に P-01 の内容に対して適用する。

### 4. 処理順序

1. 「投稿する」ボタンおよびその他の入力コントロールを無効化し、二重投稿防止機能（G-007）を作動させる。
2. P-01 の内容に対して日付プレースホルダを置換する（F-013）。
3. 画像フォルダパスから画像一覧を取得する（F-004）。
4. 画像添付対象を判定する（F-005）。
5. 投稿前チェックを行う（F-006）。
6. Misskey に投稿する（F-007）。
7. Misskey投稿が成功した場合、X投稿画面を起動する（F-008）。
8. 添付画像が1枚以上ある場合、1枚目をクリップボードにコピーする（01書 第5.5節と同じ）。
9. 画像削除事前設定機能（G-004）を実行する。
10. 各段階の結果・エラーを結果ログ（G-006）に出力する。
11. 入力コントロールを再度有効化する（成功・失敗を問わず）。

### 5. エラー時の扱い

いずれかの段階でエラーが発生した場合、CLIの終了コード相当の扱い（処理中断）を行うが、GUIではプロセスを終了しない。該当エラーをG-006で表示し、以降の段階へ進まない。ウィンドウは開いたままとする。

Misskey投稿が失敗した場合、X投稿画面は開かない（01書 第10.3節と同じ）。

X投稿画面起動が失敗した場合、Misskey投稿の取り消しは行わない（01書 第10.4節と同じ）。

### 6. 複数回投稿

GUIはCLIと異なり、1回の投稿処理後もウィンドウを閉じない。ユーザーはP-01・P-04・P-07の内容を編集し、再度「投稿する」を押下できる。結果ログは投稿ごとに追記され、自動的にはクリアされない。

### 7. エラー条件

エラー条件は、01〜02書で定義された各機能（F-002・F-004〜F-008）のエラー条件と同一とする。

### 8. 出力

処理結果・エラーを結果表示機能（G-006）へ渡す。

## G-006 GUI結果表示機能

### 1. 目的

投稿処理の結果・エラーを画面上に表示する。CLIの結果出力機能（02書 F-010）のGUI版に相当する。

### 2. 表示先

P-09 のログ表示領域に、時系列で追記する。

### 3. 表示内容

CLIの結果出力機能（F-010）と同じ情報を表示する。

| 種別 | 内容 | 表示色 |
|---|---|---|
| 正常メッセージ | 投稿テキスト取得、添付画像数、Misskey投稿結果、X投稿画面起動結果、クリップボードコピー結果、画像削除結果 | 通常色／成功メッセージは緑 |
| エラーメッセージ | エラー種別・エラー内容・対象ファイルまたは対象処理 | 赤 |

### 4. 出力しない情報

F-010 第5節と同じく、以下は表示しない。

- Misskey アクセストークン
- 設定ファイル全文
- APIレスポンスに含まれる機密情報

### 5. ログのクリア

「ログをクリア」ボタン押下時、P-09 の表示内容をすべて消去する。投稿処理の結果には影響しない。

### 6. エラー条件

本機能自体はエラーを発生させない。

### 7. 出力

なし（画面表示のみ）。

## G-007 GUI二重投稿防止機能

### 1. 目的

投稿処理の実行中に、同一内容または別内容の投稿が重複して実行されないようにする。

### 2. 処理

1. 「投稿する」ボタン押下時、P-08 および他の入力コントロール（P-01・P-03・P-04・P-05・P-06・P-07）を無効化する。
2. 投稿処理（G-005）が完了した時点（成功・失敗を問わない）で、無効化した入力コントロールを再度有効化する。

### 3. エラー条件

本機能自体はエラーを発生させない。

### 4. 出力

なし（画面制御のみ）。

### 2. 処理

GUI起動時、P-11 に `IchiPos <バージョン番号>` を表示する（02書 F-012 の出力例と同一形式）。

### 3. エラー条件

本機能自体はエラーを発生させない。

### 4. 出力

## G-009 GUI設定読み込みエラー時機能

### 1. 目的

GUI起動時に設定ファイル（`config.yaml`）の読み込みに失敗した場合の動作を定義する。

### 2. 処理

1. GUI起動時、CLIと同じ設定読み込み処理（03書 第19節）を行う。
2. 設定読み込みに成功した場合、通常通り画面を表示する。
3. 設定読み込みに失敗した場合、エラー内容を示すダイアログを表示し、ユーザーがダイアログを閉じた後、アプリケーションを終了する。

### 3. GUI起動後の再読み込み

v2では、GUI起動後に設定ファイルを再読み込みする機能は持たない。設定ファイルを修正した場合は、アプリケーションを再起動する。

### 4. エラー条件

01書・03書で定義された設定ファイルのエラー条件（設定ファイル不存在等）と同一とする。

### 5. 出力

設定エラーの内容をダイアログに表示する。アクセストークン等の機密情報は表示しない（F-010第5節と同じ方針）。

## 7. 正常系シナリオ（GUI）

### 7.1 テキスト直接入力・画像なし投稿

1. ユーザーが P-01 に投稿内容を直接入力する。
2. P-04 は未選択（空）のままとする。
3. 「投稿する」を押下する。
4. 画像添付なしとして投稿前チェックを行う。
5. Misskeyに投稿する。
6. X投稿画面を開く。
7. 結果ログに各段階の成功メッセージが表示される。

### 7.2 ファイル読み込み・画像付き投稿

1. 「ファイルから読み込む...」で `.txt` ファイルを選択し、P-01 に内容が読み込まれる。
2. 「参照...」で画像フォルダを選択する。
3. 「投稿後に画像を削除する」をオンにする。
4. 「投稿する」を押下する。
5. 画像がMisskey Driveへアップロードされ、画像付きノートが作成される。
6. X投稿画面が開き、1枚目の画像がクリップボードにコピーされる。
7. 添付画像が削除され、結果ログに削除完了メッセージが表示される。

### 7.3 日付プレースホルダを含む投稿

1. P-01 に `今日は{date}です` と入力する。
2. 「投稿する」を押下する。
3. 投稿実行時に `{date}` が実行日（`YYYY/MM/DD`形式）に置換される。
4. 置換後のテキストで投稿前チェック・Misskey投稿・X投稿画面起動が行われる。

### 7.4 同一セッションでの複数回投稿

1. 1回目の投稿を実行し、成功する。
2. ウィンドウを閉じずに、P-01 の内容を書き換える。
3. 2回目の「投稿する」を押下し、投稿する。
4. 結果ログに1回目・2回目双方の結果が時系列で表示される。

## 8. 異常系シナリオ（GUI）

### 8.1 投稿内容が空

1. P-01 が空の状態で「投稿する」を押下する。
2. 投稿前チェック（F-006）でエラーとなる。
3. Misskey投稿・X投稿画面起動を行わない。
4. 結果ログにエラー内容が表示される。
5. ウィンドウは開いたままとなり、入力コントロールは再度有効化される。

### 8.2 画像フォルダが存在しない

（GUIのフォルダ選択ダイアログでは通常発生しないが、選択後に外部でフォルダが削除された場合を想定）

1. 投稿実行時、画像フォルダ読み込み（F-004）でエラーとなる。
2. Misskey投稿・X投稿画面起動を行わない。
3. 結果ログにエラー内容が表示される。

### 8.3 Misskey投稿失敗

1. 「投稿する」押下後、Misskey投稿がエラー応答となる。
2. X投稿画面を開かない。
3. 結果ログにエラー内容が表示される。
4. 入力コントロールが再度有効化され、ユーザーは内容を修正して再投稿できる。

### 8.4 設定ファイル読み込み失敗

1. GUIモードで起動する。
2. 設定ファイル読み込みに失敗する。
3. エラーダイアログを表示する。
4. ダイアログを閉じるとアプリケーションが終了する。

## 9. 構造・実装方針

### 9.1 既存構造との関係

GUI追加後も、03書 第4節で定義された5段階の処理構造（入力受付・入力解決・投稿前チェック・投稿実行・結果出力）を維持する。

CLIとGUIで異なるのは「入力受付」と「結果出力」の実装のみとする。

| 段階 | CLI | GUI |
|---|---|---|
| 入力受付 | コマンドライン引数解析（F-001） | 画面項目からの入力（G-002〜G-004） |
| 入力解決〜投稿実行 | F-002〜F-009（共通処理） | 同左（共通処理をそのまま利用） |
| 結果出力 | 標準出力・標準エラー（F-010） | 画面ログ表示（G-006） |

投稿前チェック・投稿実行・投稿先固有処理（Misskey/X）の内部構造・責務分離方針は、03書 第7節・第8節〜第18節の方針をそのまま適用し、変更しない。

### 9.2 実行ファイル構成

GUIは、CLIと同一の実行ファイルとして提供する。実行ファイル・プロジェクトを分割しない。

実行ファイルはコンソールサブシステムの実行ファイルとする（現行のCLIと同じ）。これにより、CLIモードで起動した場合は従来通り呼び出し元のコンソールに標準出力・標準エラーが出力される。

GUIモードで起動した場合は、コンソールウィンドウを表示しないようにする（起動直後にコンソールウィンドウを非表示化する）。

### 9.3 入力受付の分離

GUIの入力受付（G-002〜G-004）は、コマンドライン引数の文字列表現を経由せず、画面項目の値を直接、入力解決以降の共通処理に渡す。これにより、GUIが実在しないコマンドライン引数を組み立てる、といった不要な変換を行わない。

投稿内容取得機能（F-002）のうち、`<content>` がファイルパスか文字列かを判定する規則は、CLI固有の入力受付規則として扱い、GUIには適用しない（詳細はG-005第3節）。

### 9.4 結果出力の分離

結果出力は、CLIでは標準出力・標準エラーへの書き込み、GUIでは画面ログへの追記として実装するが、出力すべき情報の種類（正常時・エラー時、機密情報を出力しないという方針を含む）はCLI・GUIで共通とする（F-010・G-006）。

### 9.5 投稿先固有処理との関係

Misskey投稿・X投稿画面起動の投稿先固有処理（03書 第18節）は、GUI追加後も各投稿先の処理領域に閉じたままとする。GUI層・入力受付層に投稿先固有の分岐を持ち込まない。

## 10. v2 対象外

v2 GUIでは以下を対象外とする。

- 設定ファイル（`config.yaml`）のGUI編集・保存
- 投稿履歴の保存・表示
- 画像のドラッグ&ドロップ
- 画像のサムネイルプレビュー
- `{date}` のリアルタイム置換プレビュー
- 複数投稿の同時実行・キューイング
- 複数ウィンドウ・複数タブ
- ダークモード等のテーマ切り替え
- UI文言の多言語化
- GUI起動後の設定ファイル再読み込み
- サブ投稿先の追加・複数管理・有効／無効切り替え（01書 第11節・03書 第17節の方針を継承）
