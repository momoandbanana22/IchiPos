# 開発ガイド

このドキュメントは IchiPos の開発者向けガイドです。

## 開発方針

- **TDD（Test Driven Development）で開発する**
  - テストを先に書き、そのテストを通過する実装を行う
  - 動作確認は使い捨てのスクリプトで済ませず、リポジトリに残るテストとして書く

- **ドキュメントに沿って実装する**
  - ドキュメントに記載されていない実装は行わない
  - 仕様変更が必要な場合は、まずドキュメントを更新する

- **ドキュメントは常に現行の実挙動と一致させる**
  - 挙動を変更したら、同じ変更のなかでドキュメントも実態に合わせる
  - 実装だけを変更してドキュメントを更新しないことは認めない

- **同じ事実を2箇所に書かない（Single Source of Truth）**
  - 事実の置き場所は1つに決め、他はそこを参照する
  - どうしても2箇所に持たざるを得ない場合は、**一致していることを自動で検証する**
  - 検証は次善策であり、そもそも導出できるなら1箇所にする方がよい（例: バージョン番号は csproj の `<Version>` を正典とし、`AppVersion.Current` はそこから導出する — #52）

### なぜ SSoT を方針として掲げるか

同じ事実が2箇所にあると、**必ず片方だけが更新される**。そして、どちらが正しいのか誰にも分からなくなる。

本プロジェクトで実際に起きた事例は、いずれも根が同じである。

| 事例 | 二重化していた事実 | 起きたこと |
|---|---|---|
| #34 / #36 | 実挙動とドキュメント | 実装だけ変更され、リリース済みバージョンの説明が誤ったまま利用者に届いた |
| #45 | バージョン番号（`AppVersion.cs` と `csproj`） | 片方だけ更新され、7リリースにわたり配布exeが誤ったバージョンを名乗った |
| #50 | 分析内容（Issue と PR 本文） | 同じ内容が2箇所に書かれ、レビュー時に重複が指摘された |

上記の「ドキュメントは常に現行の実挙動と一致させる」も、本質的には SSoT を保てない領域（ドキュメントは実装から自動導出できない）に対する運用でのカバーである。
自動導出できる場所では、運用でのカバーに頼らず1箇所にすること。

### 情報の正典（どこに何を書くか）

同じ事実を2箇所に書かないために、情報の種類ごとに**正典（唯一の置き場所）を1つ**定める。正典以外の場所には内容を書かず、**正典への参照（リンク）を置く**。参照で到達性が確保できない箇所（PRテンプレートのチェックリスト等）に限り複製を許し、その場合は一致を検証する手段をあわせて用意する。

| 情報の種類 | 正典（唯一の置き場所） | 正典以外の場所での扱い |
|---|---|---|
| 開発プロセス・ルール（TDD・ドキュメント駆動・SSoT・Git/ブランチ運用・リリース手順・実装前後の確認） | この `DEVELOPMENT.md` | `CLAUDE.md`・`.github/pull_request_template.md` は内容を書かず該当節を参照する（→ #60） |
| AI エージェント固有の行動規範 | [`CLAUDE.md`](./CLAUDE.md) | 他所に複製しない |
| CLI の仕様・挙動（技術契約） | [`doc/01〜03`](./doc/) | README は利用者向けに言い換える。挙動の技術的定義（値・書式・上限等）は doc を正典とし、README や他文書はそれを参照する |
| GUI の仕様・挙動（技術契約） | [`doc/04`](./doc/04_IchiPos_GUI仕様書.md) | 同上 |
| 利用者向けの使い方・操作説明 | [`README.md`](./README.md) | 使い方・操作を説明する。挙動の技術的定義（値・書式等）そのものは doc が正典で、README はそれを参照する |
| 設定項目 | [`config.yaml.example`](./config/config.yaml.example) | README からは参照する |
| 個別インシデントの記録（何が起きたか・調査結果） | 起票した Issue | 他所へ転記しない。参照は Issue 番号で行う |
| 失敗から得た教訓・ルールの根拠（なぜこのルールがあるか） | この `DEVELOPMENT.md` の「なぜ」節 | Issue の要約ではなく教訓として維持する。個別事例の詳細は Issue を参照する |
| 配布物・リリースに関する横断的な告知 | 該当告知を1箇所（原則 README の該当節）に書く | 各リリースノートからはリンクし、全リリースへ複製しない。過去に個別複製したものは歴史的記録として残す |

この表自体も正典であり、複製の解消は段階的に進める。上位の課題と経緯は #55（および分割 Issue #60〜#63）を参照。

## 意思決定と優先順位

### 残件の優先順位

残件・タスクの順位は、**記録された実害・プロジェクトへの価値**で決める。着手しやすさ（自分だけで完結できるか、直しやすいか）を順序の理由にしない。

過去に、残件の優先順位を「単独で完結できるか」で並べ、最も些末な項目を先頭に推してしまった（バージョン導出 #52 を SSoT #55 より上に置いた）。実際に繰り返し損害が出ていたのは情報の重複（#34 / #36 の doc 乖離、リリースノートの複製）であり、そちらが上位だった。低リスクで単独完結できる項目を「今すぐ動かせる候補」として別枠で添えるのは可だが、それを最優先として提示しない。

### 方針・アーキの選択（意思決定）

複数の有効な選択肢がある方針・アーキテクチャの決定は、思いつきや「おすすめ」で進めず、**構造化して評価する（CMMI DAR: Decision Analysis and Resolution）**。

1. 評価基準を定め、重みを付ける（基準の抜け・二重計上に注意）
2. 各案を基準ごとに採点し、加重スコアで比較する
3. 感度（重みを変えると順位がどう動くか）も確認する
4. 推奨は述べてよいが、**最終選択は決定権者（オーナー）が行う**。評価と決定の記録を Issue 等に残す

## 開発環境

- OS：Windows 11
- 言語：C#
- フレームワーク：.NET 10
- アプリ形式：コンソールアプリケーション（CLI）+ WPF（GUI）。実行ファイルは1つで、起動時の引数の有無により自動的に切り替わる（引数なし→GUI、引数あり→CLI）

## 実装前の確認

実装を開始する前に、以下を確認してください：

1. `gh issue list` でオープンな Issue を確認する（着手しようとしている内容が既に起票されていないか）
2. 関連する設計書を読む
3. ドキュメントに記載された仕様を理解する
4. ドキュメントに記載されていない実装は行わない

## 作業中に見つけたことの扱い

**追跡できない場所に書き残さない。** チャットでの報告、PR 本文、Issue のコメントは、いずれも後から読み返されない。

1. 本題と別の問題に気づいたら、**その場で `gh issue create` する**。報告や本文に書くだけで済ませない
2. 未対応の作業を「必要であれば起票してください」と書き残さない。**気づいた側が起票する**
3. **Issue をクローズする前に、未対応事項が残っていないか確認する。** 残っていれば別 Issue を起票してからクローズする

クローズ済み Issue のコメントは誰も見返さない。そこに残した作業は失われる。

### なぜこの節があるか

過去に、調査結果・気づいた不具合・未対応の判断を、チャットやクローズ済み Issue のコメントに書いて放置した事例が1日に4回発生した（issue #50）。
いずれもレビュー時の指摘がなければ失われていた。うち1件は、既存 Issue の存在に気づかず同じ内容を重複実装したものだった（#39）。

## 実装後の確認

PR を作成する前に、以下を確認してください：

1. **`.github/pull_request_template.md` を読み、その項目を PR 本文に反映する**
2. **変更した挙動が、上記のドキュメントすべてと一致しているか確認する**
3. 一致していないものがあれば、同じ PR のなかでドキュメントを実態に合わせる
4. ドキュメントの更新が不要と判断した場合は、その理由を PR に書く

> **`gh pr create --body` はテンプレートを迂回します。**
> PR テンプレートは Web UI での作成時、または `gh pr create` で本文を指定しなかった場合にのみ本文の初期値として挿入されます。
> `--body` を指定するとテンプレートは一切適用されないため、CLI で PR を作成する場合はテンプレートを自分で読んで反映してください。
> 過去に、この迂回によりテンプレート内のチェックリストとルールが3つの PR で適用されなかった事例があります（issue #50）。

### 「仕様変更ではない」変更にも適用される

この確認は、`feat:` だけでなく **`fix:` や `refactor:` にも適用される**。

過去に、バグ修正としてクリップボードへのコピー枚数を変更した際、コードとテストのみを変更してドキュメントを一切更新しなかったことがある（issue #34）。その結果、リリース済みバージョンの説明が誤ったまま利用者に届き、正しい挙動を記述した文書が1つも存在しない状態になった。

**利用者から見た挙動が変わるなら、それは仕様変更である。** コミットの種別で判断しないこと。

### ドキュメント更新が不要なケース

以下は更新不要と判断してよい。

- 内部実装のみの変更で、利用者から見た挙動・画面・設定・ログ出力が一切変わらないリファクタリング
- テストの追加・修正のみ
- ビルド設定・CI設定のみの変更

判断に迷う場合は更新する側に倒すこと。

## Git / ブランチ運用

- **マージ済みのブランチを削除しない。** feature / fix / docs ブランチは PR マージ後も `origin` に残す。過去のマージ済み PR のブランチがすべて `origin` に残っている慣習に合わせること。削除が必要と判断した場合は、実行せず先に確認する。
- 破壊的・外向きの Git 操作（ブランチ・タグの削除、force push）は、`git log` や既存ブランチなどリポジトリの実際の状態から慣習を確認したうえで行う。自分の「ふつうこうする」という既定を当てない。
- **`main` はブランチ保護されている。** PR は CI（`ci.yml` の `test` ジョブ）が成功しないとマージできない（#49）。管理者への強制（enforce_admins）は無効にしてあり、これはリリース手順でバージョン更新の chore コミットを `main` へ直接 push するため。**enforce_admins を有効にするとこの直 push が塞がり、リリース手順が破綻する。** 変更する場合はリリース手順もあわせて見直すこと。

## リリース手順

バージョン番号の正典は `IchiPos.Core/IchiPos.Core.csproj` の `<Version>` **1箇所**。リリース時はここだけを更新する（#52）。

| 場所 | 用途 |
|---|---|
| `IchiPos.Core/IchiPos.Core.csproj` の `<Version>` | 配布する `IchiPos.exe` のファイルバージョン。**バージョン番号の正典** |

`--version` の出力・GUIのバージョン表示（P-11）で使う `AppVersion.Current` は、この `<Version>` から生成されるアセンブリ属性を実行時に読み取って導出する。手で更新する必要はない。

手順は以下のとおり。

1. csproj の `<Version>` を新しいバージョン番号に更新し、`chore: バージョンを X.Y.Z に更新` としてmainへコミットする
2. `vX.Y.Z` のタグを push する
3. `.github/workflows/release.yml` がテスト・publish・zip化・リリース作成まで自動実行する

### 横断的な告知（配布物の問題など）

複数リリースに関わる告知（配布物の不備など）は、全リリースのノートへ複製せず、正典1箇所に書いて各リリースノートからリンクする（書き方は「[情報の正典（どこに何を書くか）](#情報の正典どこに何を書くか)」の表に従う）。過去に個別複製した告知は歴史的記録として残す。

### リリースが失敗した・実行されなかった場合

`release.yml` は `workflow_dispatch` にも対応しているため、**タグを打ち直さずに再実行できる**（issue #41）。

```
gh workflow run release.yml --ref vX.Y.Z
```

`--ref` にはタグを指定する。ブランチを指定した場合は「Verify version matches tag」で停止する（ブランチ名でリリースが作られるのを防ぐため。#54 で実機確認済み）。

既にリリースが作成済みの状態で再実行した場合の挙動は #54 で実測した。

- 既存のリリースを**その場で更新**する（新規作成や重複はしない）。同名の zip アセットを差し替える。
- ただし `generate_release_notes: true` により、**自動生成の「What's Changed」がノートに追記される**。手で追記した注意書き等は消えない（上書きはされない）が、再実行のたびに changelog が重複して蓄積する。**再実行したら、重複した「What's Changed」節を手で取り除くこと。**

> **v2.3.1 以前のタグは手動実行できません。**
> `workflow_dispatch` は指定した ref にあるワークフローファイルを実行するため、この対応より前のタグには手動実行の口がありません。
> それらをやり直す必要が生じた場合は、タグの打ち直しが必要です。

### 更新漏れ・取り違えの検出

過去に `csproj` 側の更新が漏れ、v2.2.2〜v2.3.0 の7リリースで配布exeが 2.2.1 と名乗る状態が続いた（issue #45）。当時はバージョン番号が2箇所にあり、片方だけ更新されたことが原因だった。**#52 で正典を csproj の `<Version>` 1箇所に集約したため、この「2箇所が食い違う」種類の取り違えは構造的に起きなくなった。**

残る「タグだけ進めてバージョンを上げ忘れる」等のずれに対して、以下2つの自動チェックがある。**手順を覚えていなくても、漏れればCIが落ちる。**

- `AppVersionTests`：csproj から導出される `AppVersion.Current` が、csproj の `<Version>` 宣言値と一致することを検証する。導出の配線（アセンブリ属性の生成・解析）が壊れた場合に落ちる。`dotnet test` が走るすべての場所（PR・リリース）で実行される
- `release.yml` の「Verify version matches tag」：push されたタグ名（`vX.Y.Z`）と csproj の `<Version>` が一致することを検証する。タグだけ進めてバージョンを上げ忘れた場合に、publish前で停止する

### ワークフロー変更の事前検証（スモークテスト・actionlint）

publish → zip → リリース作成の手順は `release-core.yml`（`workflow_call` の共通ワークフロー）に集約している。本番リリース（`release.yml`）と試走（`release-smoke-test.yml`）が同じ手順を共有するため、試走で通った手順がそのまま本番でも使われる（SSoT）。

`.github/workflows/**` を変更する PR では、以下が自動で走る。

- `release-smoke-test.yml`：`release-core.yml` を **dry_run（ドラフトリリース）で試走**し、`action-gh-release` 等のリリース公開の仕組みが壊れていないかを**実リリースを公開せずに**検証する。作成したドラフトはアセット名を検証したうえで削除する。
- `actionlint`：ワークフロー YAML の構文・古い/誤った action 記法・run スクリプトの誤りを静的検査する。

これにより、リリース周りの変更を「次の実リリース」を待たずにマージ前へ検証できる（#76 / #77）。ただし本番のみのステップ（`Verify version matches tag`、`draft:false`）はスモークテストでは走らないため、そこは従来どおり実リリース時に確認する。
