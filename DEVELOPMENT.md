# 開発ガイド

このドキュメントは IchiPos の開発者向けガイドです。

## 開発方針

- **TDD（Test Driven Development）で開発する**
  - テストを先に書き、そのテストを通過する実装を行う

- **ドキュメントに沿って実装する**
  - ドキュメントに記載されていない実装は行わない
  - 仕様変更が必要な場合は、まずドキュメントを更新する

- **ドキュメントは常に現行の実挙動と一致させる**
  - 挙動を変更したら、同じ変更のなかでドキュメントも実態に合わせる
  - 実装だけを変更してドキュメントを更新しないことは認めない

## 開発環境

- OS：Windows 11
- 言語：C#
- フレームワーク：.NET 10
- アプリ形式：コンソールアプリケーション（CLI）+ WPF（GUI）。実行ファイルは1つで、起動時の引数の有無により自動的に切り替わる（引数なし→GUI、引数あり→CLI）

## 実装前の確認

実装を開始する前に、以下を確認してください：

1. `gh issue list` でオープンな Issue を確認する（着手しようとしている内容が既に起票されていないか）
2. 関連する設計書を読む
3. ドキュメントに記載された仕様を理解する
4. ドキュメントに記載されていない実装は行わない

## 作業中に見つけたことの扱い

**追跡できない場所に書き残さない。** チャットでの報告、PR 本文、Issue のコメントは、いずれも後から読み返されない。

1. 本題と別の問題に気づいたら、**その場で `gh issue create` する**。報告や本文に書くだけで済ませない
2. 未対応の作業を「必要であれば起票してください」と書き残さない。**気づいた側が起票する**
3. **Issue をクローズする前に、未対応事項が残っていないか確認する。** 残っていれば別 Issue を起票してからクローズする

クローズ済み Issue のコメントは誰も見返さない。そこに残した作業は失われる。

### なぜこの節があるか

過去に、調査結果・気づいた不具合・未対応の判断を、チャットやクローズ済み Issue のコメントに書いて放置した事例が1日に4回発生した（issue #50）。
いずれもレビュー時の指摘がなければ失われていた。うち1件は、既存 Issue の存在に気づかず同じ内容を重複実装したものだった（#39）。

## 実装後の確認

PR を作成する前に、以下を確認してください：

1. **`.github/pull_request_template.md` を読み、その項目を PR 本文に反映する**
2. **変更した挙動が、上記のドキュメントすべてと一致しているか確認する**
3. 一致していないものがあれば、同じ PR のなかでドキュメントを実態に合わせる
4. ドキュメントの更新が不要と判断した場合は、その理由を PR に書く

> **`gh pr create --body` はテンプレートを迂回します。**
> PR テンプレートは Web UI での作成時、または `gh pr create` で本文を指定しなかった場合にのみ本文の初期値として挿入されます。
> `--body` を指定するとテンプレートは一切適用されないため、CLI で PR を作成する場合はテンプレートを自分で読んで反映してください。
> 過去に、この迂回によりテンプレート内のチェックリストとルールが3つの PR で適用されなかった事例があります（issue #50）。

### 「仕様変更ではない」変更にも適用される

この確認は、`feat:` だけでなく **`fix:` や `refactor:` にも適用される**。

過去に、バグ修正としてクリップボードへのコピー枚数を変更した際、コードとテストのみを変更してドキュメントを一切更新しなかったことがある（issue #34）。その結果、リリース済みバージョンの説明が誤ったまま利用者に届き、正しい挙動を記述した文書が1つも存在しない状態になった。

**利用者から見た挙動が変わるなら、それは仕様変更である。** コミットの種別で判断しないこと。

### ドキュメント更新が不要なケース

以下は更新不要と判断してよい。

- 内部実装のみの変更で、利用者から見た挙動・画面・設定・ログ出力が一切変わらないリファクタリング
- テストの追加・修正のみ
- ビルド設定・CI設定のみの変更

判断に迷う場合は更新する側に倒すこと。

## リリース手順

バージョン番号は以下の**2箇所**にあり、両方を同じ値に更新する。

| 場所 | 用途 |
|---|---|
| `IchiPos.Core/AppVersion.cs` の `AppVersion.Current` | `--version` の出力、GUIのバージョン表示（P-11） |
| `IchiPos.Core/IchiPos.Core.csproj` の `<Version>` | 配布する `IchiPos.exe` のファイルバージョン |

手順は以下のとおり。

1. 上記2箇所を新しいバージョン番号に更新し、`chore: バージョンを X.Y.Z に更新` としてmainへコミットする
2. `vX.Y.Z` のタグを push する
3. `.github/workflows/release.yml` がテスト・publish・zip化・リリース作成まで自動実行する

### 更新漏れの検出

過去に `csproj` 側の更新が漏れ、v2.2.2〜v2.3.0 の7リリースで配布exeが 2.2.1 と名乗る状態が続いた（issue #45）。
再発を防ぐため、以下2つの自動チェックがある。**手順を覚えていなくても、漏れればCIが落ちる。**

- `AppVersionTests`：`AppVersion.Current` と `csproj` の `<Version>` が一致することを検証する。`dotnet test` が走るすべての場所（PR・リリース）で実行される
- `release.yml` の「Verify version matches tag」：push されたタグ名（`vX.Y.Z`）とバージョン番号が一致することを検証する。タグだけ進めてバージョンを上げ忘れた場合に、publish前で停止する
