# プロジェクトルール

このプロジェクトで作業する際は、以下のルールに従ってください。

## 開発方針

1. **TDD（Test Driven Development）で開発する**
   - テストを先に書き、そのテストを通過する実装を行う

2. **ドキュメントに沿って実装する**
   - 必ず DEVELOPMENT.md を最初に確認してください
   - 基本仕様書（`doc/01_IchiPos_基本仕様書.md`）を参照
   - 機能設計書（`doc/02_IchiPos_機能設計書.md`）を参照
   - 構造設計書（`doc/03_IchiPos_構造設計書.md`）を参照
   - GUI仕様書（`doc/04_IchiPos_GUI仕様書.md`）を参照（GUI関連の実装時）
   - ドキュメントに記載されていない実装は行わない

3. **ドキュメントは常に現行の実挙動と一致させる**
   - 挙動を変更したら、同じ変更のなかでドキュメントも実態に合わせる
   - 実装だけを変更してドキュメントを更新しないことは認めない

## 実装前の確認

実装を開始する前に、以下を確認してください：

1. DEVELOPMENT.md を読む
2. 関連する設計書を読む
3. ドキュメントに記載された仕様を理解する

## 実装後の確認

PR を作成する前に、以下を確認してください：

1. 変更した挙動が、以下のすべてと一致しているか確認する
   - `doc/01_IchiPos_基本仕様書.md`（CLIの仕様・挙動）
   - `doc/02_IchiPos_機能設計書.md`（CLIの仕様・挙動）
   - `doc/03_IchiPos_構造設計書.md`（処理構造・責務分担）
   - `doc/04_IchiPos_GUI仕様書.md`（GUIの画面・機能）
   - `README.md`（**利用者向けの説明。忘れやすいので特に注意**）
   - `config/config.yaml.example`（設定項目）
2. 一致していないものがあれば、同じ PR のなかでドキュメントを実態に合わせる
3. ドキュメントの更新が不要と判断した場合は、その理由を PR に書く

**この確認は `feat:` だけでなく `fix:` や `refactor:` にも適用されます。** 利用者から見た挙動が変わるなら、コミットの種別にかかわらず仕様変更として扱ってください。過去に、バグ修正としてコードとテストのみを変更しドキュメントを放置した結果、リリース済みバージョンの説明が誤ったまま利用者に届いた事例があります（issue #34）。

内部実装のみの変更（利用者から見た挙動・画面・設定・ログ出力が一切変わらないリファクタリング、テストのみの変更、ビルド/CI設定のみの変更）は更新不要です。迷う場合は更新する側に倒してください。
